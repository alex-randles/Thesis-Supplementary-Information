Introduction back to ToC

The ontology extends the LODE ontology [LODE] which is designed to represent Linked data events. The design has been inspired by the state of the art in Linked data change detection [DSNotify]. Fig.1 shows the class interaction diagram for the ontology.


The detection of changes within the source data is initiated by an agent who is responsible for maintaining the data (cdo:hasMaintainer). Changes will be detected between a previous version (cdo:hasPreviousSource) and the current version (cdo:hasCurrentversion). Once the change detection process has been initiated, changes are detected between when the process began (cdo:hasDetectionStart) and a predetermined end date (cdo:hasDetectionEnd) which is defined by the maintainer. A notification policy (cdo:hasNotificationPolicy) can be defined which is represented with the Rei policy ontology. The notification policy allows agents to be notified of changes when a certain amount has occurred. Therefore, ensuring the maintainers are informed of changes when required.

Changes which occur in a specific source data are grouped into a log (cdo:ChangeLog). Each change (cdo:hasChange within a change log is represented as a specialized event (lode:Event) which has resulted in a change in the source data used to generate linked data. Different change types exist within the ontology which allow the cause of the change to be captured. For instance, a value is inserted (cdo:InsertSourceData) in the source data. Additional information related to the change such as the data inserted (cdo:hasChangedData) or its location within the source data (cdo:hasDataReference). The model can be extended to capture domain specific changes.
